# 2026-06-16 — Auth API Implementation

## Session 1 — Core Auth (Register / Login / Logout / Refresh / VerifyEmail / ResendVerification)

1. Implemented the auth module using Repository Pattern (userRepo, authRepo) with service layer separation.
2. Access tokens are short-lived JWTs (15 min, HS256 signed with APP_SECRET). JWT includes `sid` (session_id) claim so logout can revoke the correct session.
3. Refresh tokens are random 32-byte hex tokens stored hashed (SHA-256) in DB, sent as HttpOnly cookie scoped to `/auth/refresh`.
4. Email verification tokens are also hashed before storage.
5. Email service is stubbed (logs token) — real provider (Resend/SendGrid) to be integrated later.

## Session 2 — Extended Auth (Password Reset / Google OAuth / 2FA / Sessions / Profile)

1. 2FA login flow: Login with password → if TOTP enabled, return `{2fa_required: true, pending_token: <jwt>}` with type=`2fa_pending`. Client calls `/auth/2fa/challenge` with pending_token + totp_code to get full tokens. Pending token is a 5-min JWT rejected by JWTAuth middleware.
2. TOTP setup is two-step: `/2fa/setup` returns secret + OTP URL (not saved), `/2fa/enable` takes secret + code to confirm and persist. Backup codes are SHA-256 hashed before DB storage and consumed one-time.
3. Google OAuth uses state cookie (10 min, HttpOnly) for CSRF protection. New OAuth users get an empty UserCredential row so downstream credential lookups don't fail.
4. Password reset revokes all sessions on completion.
5. `DELETE /auth/sessions` revokes all sessions except the current one (identified by `sid` JWT claim).
6. Added `APP_URL`, `APP_FRONTEND_URL`, `GOOGLE_CLIENT_ID`, `GOOGLE_CLIENT_SECRET` to config.
